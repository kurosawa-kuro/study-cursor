## 2. 権威順位（矛盾したときの勝者）

```
01_仕様と設計.md  >  03_実装カタログ.md  >  README.md
```

`05_運用.md` / `04_検証.md` / `02_移行ロードマップ.md` / `教育資料/*` は上記から派生する従属ドキュメント。
`AGENTS.md` は AI コーディングアシスタント向けの作業ガイドで、仕様の権威ではない（仕様変更があれば `01` を直してから `AGENTS.md` に反映する）。

## 3. 更新ルール（変更種別 → 直すドキュメント）

| 変更 | 連動して直すドキュメント |
|---|---|
| 想定ユーザー / ゴール / スコープの変更 | `01_仕様と設計.md` → `README.md` |
| `src/office-task/README.md` のプロンプト追加・削除 | `03_実装カタログ.md §2` + （必要なら）`05_運用.md §3` |
| `src/office-task/data/` の追加・削除 | `03_実装カタログ.md §2` |
| 設計上の決め事の変更（例: 元ファイル不変更の方針、ランタイム非インストール原則、Hobby 枠運用前提） | `01_仕様と設計.md §5` + `AGENTS.md` + `src/office-task/README.md` |
| Cursor の UI / Hobby 枠仕様の変更 | `教育資料/04_Cursor操作.md` + `教育資料/01_クイックスタート.md` |
| 新候補（追加したいプロンプト・ドキュメント）の発生 | `02_移行ロードマップ.md` |

**原則**: 連動するドキュメントは **同一コミット／PR で直す**。後回しにすると drift する。

## 4. 書き方ルール

- **日本語で書く**。英単語はそのまま OK（例: `Cursor`, `PowerShell`, `Excel`）
- **節番号を本文に対応させる**（`§1`, `§1.2` 形式）。他ドキュメントからの相互参照を壊さない
- **手順はコピペで動く形**で書く。プレースホルダ（`<...>`）は配布元から取得する値に限定する
- **前提を伏せない** — 「Cursor インストール済み」「Open Folder 済み」など、必要な前提を冒頭で宣言する
- **推測で書かない**。手順を書いたら実際に Cursor で確認する

---

## 5. ドキュメントを追加する前に

1. 既存ドキュメント（`01`〜`05` + `教育資料/00`〜`03`）の役割と被らないか確認。被るなら追加せず既存を拡張する
2. 追加時は本ファイル §1 に行を追加し、[`../README.md`](../README.md) のドキュメント一覧にもリンクを追加
3. 権威順位（§2）の位置付けを決める（従属 / 並列）
4. 更新契機（§3）を表に追加する

---

## 6. メンテナンス観点

- `03_実装カタログ.md` は放置すると実態と乖離する。`src/` を変更したら必ず同時更新
- `04_検証.md` の「よくある問題」は利用者からの報告ベースで継続的に育てる
- `02_移行ロードマップ.md` は **未着手のものだけ** を残す。実装済みは削除して履歴は git log に任せる
- `教育資料/04_Cursor操作.md` の Hobby 枠の数値（slow request 上限など）は Cursor 公式 docs と乖離しがちなので、定期的に確認して更新する
